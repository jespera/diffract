# Grammar metadata pipeline

## 1. Purpose

Each tree-sitter grammar ships a `node-types.json` file alongside its
parser, generated by `tree-sitter generate`. This file is the
authoritative description of the grammar's node types: which fields
they have, which child types they accept, which supertypes they
participate in. The metadata is needed by several diffract features
that benefit from grammar awareness — role-aware metavariable naming,
type-restricted holes, field-aware error messages, anti-unification
weighting, and bracket/string/comment information for any future
matcher that operates at the token level.

This pipeline brings that metadata into diffract at build time,
parses it lazily, and exposes typed accessors via a small OCaml
module. It is independent of any specific consumer: it lays the
foundation that consumers can be built on.

## 2. Pipeline

### 2.1 Build-time extraction

`grammars/build-grammars.sh` copies each tree-sitter package's
`node-types.json` into `grammars/metadata/<language>.node-types.json`
after `npm install`. Files are committed alongside the grammar
binaries so they version with the grammar. Five languages currently:
`typescript`, `tsx`, `kotlin`, `php` (php_only), `scala`.

### 2.2 Build-time embedding

`lib/dune` inlines each metadata file's bytes as a raw OCaml string
constant in a generated `embedded_metadata.ml` (auto-generated, not
committed). The generated module exposes one string per language and
a `by_language : string -> string option` lookup. The build pulls JSON
content into the binary; the runtime sees string literals.

No hand-edited generated `.ml` files are produced — only data is
embedded. JSON files in `grammars/metadata/` remain the single source
of truth.

### 2.3 Runtime parsing and lookup

`lib/grammar_metadata.ml{,i}` parses each language's embedded JSON
lazily on first lookup and memoizes the result. The current API
exposes the DEL definition — the bracket pairs, string-literal shapes,
and comment markers derived from `grammar.json` — plus the language
list:

```ocaml
type string_def = { opener : string; closer : string; escape : char option }
type del_definition = {
  bracket_pairs : (string * string) list;
  string_defs : string_def list;
  line_comments : string list;
  block_comments : (string * string) list;
}
val del_definition : language:string -> del_definition
val all_languages : unit -> string list
```

The API is deliberately narrow. As consumers materialise — role-aware
metavar naming, type-restricted holes — additional accessors will be
added that read different projections out of the same parsed JSON.

## 3. History: the `unwrap_root` consumer

The pipeline was originally scoped to also feed `Tree.unwrap_root` in
`lib/tree.ml`: a `list_shape_wrappers` / `is_list_shape_wrapper`
projection over `node-types.json` (named nodes with no field-named
children) supplied candidate wrapper types, and `unwrap_root` peeled
them at the root of a parsed pattern fragment so a single-node pattern
matched against the corresponding node in the source rather than the
whole file.

The universal tokenizer made that consumer unnecessary: the pattern is
tokenized to a leaf stream rather than parsed as a fragment, so there
is no root to peel. `Tree.unwrap_root` and the `list_shape_wrappers`
projection were both removed (verified: a single-import pattern matches
and rewrites the right line in a multi-import file with no root peel).
The DEL accessors above are the part of the pipeline the new
architecture retains.

## 4. Trade-offs

### 4.1 Pros

- Adding a new language to diffract is mechanical: register the
  grammar binding, drop in `node-types.json`, the metadata pipeline
  picks it up.
- Grammar metadata stays in sync with the grammar version; no manual
  curation drift.
- Single binary distribution preserved; the JSON files are inlined
  at build time, not loaded from disk at runtime.
- The pipeline is consumer-agnostic. New accessors for new use cases
  read the same parsed JSON.

### 4.2 Cons

- One new build dependency (yojson) and one new OCaml module.
- `node-types.json` files are committed (a few hundred KB total) and
  must be regenerated when a grammar is upgraded — `build-grammars.sh`
  does this automatically when run.
- Schema drift in `node-types.json` upstream would need the parser to
  update. The schema is stable in practice but not formally
  versioned.
- The live API exposes only the DEL definition accessors. The
  `node-types.json` projection has no consumer since `unwrap_root` was
  removed (§3); the embedding is retained as foundation for future
  field-info / supertype use cases but is currently unexercised.

## 5. Open questions

- **What's the minimal API for the hybrid matcher to consume?**
  Likely needs field-info per node type, supertype/subtype lookup,
  and bracket-pair extraction — none of which the current API
  exposes. These will be added when the matcher work begins.
- **Should bracket-pair / string-delimiter / comment-marker info be
  extracted from `node-types.json` or from the grammar source?**
  `node-types.json` describes node-type structure but not anonymous
  tokens; the grammar source (`grammar.js`) has the lexer rules but
  is harder to parse mechanically. Worth deciding when the consumer
  needs this information.
- **Should the smoke test be promoted to a startup invariant?** The
  current implementation is a unit test; an alternative is for
  `Context.create` (or another startup hook) to validate that every
  registered language has a non-empty DEL definition and fail fast if
  not. Defer until we have a consumer where stale-metadata behaviour
  would otherwise be silent.
